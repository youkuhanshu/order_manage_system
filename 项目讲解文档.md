# 第 1 部分　项目全景

## 1.1 这是个什么程序（用大白话说）

这是一个**单机版桌面点餐软件**，界面模仿「美团 / 饿了么」。一个完整的使用流程是：

1. 打开程序 → 看到**登录页**，输入用户名密码登录（或注册新账号）。
2. 进入**菜单页**：左边是分类（热菜、凉菜、饮品……），右边是一张张菜品卡片。
3. 点卡片上的「+」把菜**加进购物车**；如果你是会员，卡片上会显示打折后的会员价。
4. 进入**购物车页**：增减数量、看实付金额（已经按会员等级打好折）。
5. 点「去结算」→ 弹出**结算窗口**：显示金额和取餐号，可以顺手给这顿饭打个星、写句评价。
6. 结算后这单进入**排队**：你会拿到一个取餐号，在**排队页**能看到「当前叫号」、「预约排队」、「待取餐」三块进度，号会随时间自动往前走。

所有数据（菜单、用户、评论、历史订单）都存在几个 **txt 文本文件**里。**没有联网、没有数据库、没有服务器，全部跑在本机。** 可以理解成「美团 App 的点餐部分，但数据存在记事本里」。

# 第 2 部分　Qt / C++ 零基础扫盲（必读）

我们组没学过 Qt，所以这一部分尽量把后面代码里会反复见到的东西讲清楚。**理解了这 11 个小节，整个项目的代码你能看懂八成。**

## 2.1 Qt 是什么？为什么用它？

C++ 本身只能写「黑框框」命令行程序，**画不出窗口、按钮**。Qt 是一套 C++ 的**图形界面库**，它提供了大量现成的「控件」（窗口、按钮、文字、输入框……），我们只要把这些控件**摆好位置、连好逻辑**，就能拼出一个像样的桌面软件。

你在屏幕上看到的每一个东西，在代码里都是一个 Qt 对象。比如一个按钮就是一个 `QPushButton` 对象。

## 2.2 控件家族（本项目用到的）

## 2.3 布局系统（UI 的骨架，重点）

Qt 里**控件不是用坐标硬摆的**，而是放进「布局（Layout）」里，由布局自动算位置、自动随窗口缩放。这是 Qt 界面的核心思想。本项目只用两种布局：

- `QVBoxLayout`：里面的控件**从上到下竖着排**。
- `QHBoxLayout`：里面的控件**从左到右横着排**。

常用操作（看代码必须认识）：

```cpp
auto *layout = new QVBoxLayout(this);   // 给 this 这块界面装一个"竖排布局"
layout->setContentsMargins(16,16,16,16);// 四周留白 16 像素（左、上、右、下）
layout->setSpacing(12);                 // 控件之间间隔 12 像素
layout->addWidget(someButton);          // 往布局里塞一个控件
layout->addLayout(anotherLayout);       // 往布局里塞另一个布局（布局可以嵌套！）
layout->addStretch();                   // 加一个"弹簧"，把后面的东西顶到一边
layout->addSpacing(10);                 // 加一段固定空白
```

**嵌套**是关键：一个横排布局里可以放一个竖排布局，竖排里又放横排……界面就是这样像搭积木一样拼出来的。比如菜品卡片 `DishCard` 的结构就是：

```
水平布局(QHBoxLayout)
├── 左：图片色块(QFrame)
├── 中：竖直布局
│      ├── 菜名 + 榜单徽章（又是一个水平布局）
│      ├── 描述
│      └── 评分 + 月售 + 查看评论（又是一个水平布局）
└── 右：竖直布局（价格 + 加购按钮）
```

**「弹簧」`addStretch()` 是新手最容易迷惑的点**：它会尽量占满剩余空间，从而把它前后的控件「推开」。比如导航栏里 `addStretch()` 放在中间，就能让左边的按钮靠左、右边的菜品数靠右。

## 2.4 信号与槽（signals & slots，全项目最重要的机制）

这是 Qt 的灵魂，也是本项目出现最多的写法。一句话：**「某件事发生了」→ 自动「去做另一件事」**。

三个名词：

- **信号（signal）**：事情发生时控件「喊一声」。比如按钮被点了，就发出内置的 `clicked` 信号。
- **槽（slot）**：收到喊声后要执行的那段代码。
- **`connect(发出者, 信号, 接收者, 要执行的代码)`**：把「谁喊」和「喊了之后做什么」绑在一起。

例子（登录页的登录按钮）：

```cpp
connect(loginBtn, &QPushButton::clicked, this, [this]() {
    emit loginClicked(m_nameEdit->text(), m_pwdEdit->text());
});
```

逐块翻译：

- `loginBtn` —— 谁会发信号：登录按钮。
- `&QPushButton::clicked` —— 哪个信号：被点击。
- `this` —— 谁来接收。
- `[this]() { ... }` —— 这是一个 **lambda（匿名函数）**，就是「收到信号后要执行的代码」。`[this]` 表示这段代码里可以用到当前对象的成员。
- `emit loginClicked(...)` —— **`emit` 表示"发射一个信号"**。这里 `loginClicked` 是我们**自己定义的信号**，把用户名密码打包喊出去，交给主窗口去验证。

**为什么要这么绕？** 这就是本项目的核心设计：**页面只负责"喊"，不负责"做"**。登录页不知道怎么验证密码，它只把账号密码喊出来；真正验证的逻辑写在主窗口。这样**界面和业务逻辑彻底分家**，改界面不影响逻辑、改逻辑不影响界面，多人协作也不容易冲突。

自定义信号怎么声明？在类的头文件里：

```cpp
signals:
    void loginClicked(const QString &name, const QString &password);
```

信号**只声明、不写实现**（Qt 的 moc 工具会自动生成），我们只管在需要时 `emit` 它。

一个进阶写法（评论弹窗里出现过）：

```cpp
connect(m_sortCombo, QOverload<int>::of(&QComboBox::currentIndexChanged),
        this, &CommentDialog::onSortChanged);
```

`QComboBox::currentIndexChanged` 有两个重载版本（一个传 `int`、一个传 `QString`），`QOverload<int>::of(...)` 是用来**明确指定"我要带 int 参数的那个版本"**。这里把「下拉框选项变了」连到「重新排序」的槽函数 `onSortChanged`。

## 2.5 父子对象与内存管理（为什么 new 了却不 delete？）

学过 C++ 的同学第一眼会担心：代码里全是 `new`，却几乎看不到 `delete`，不会内存泄漏吗？

**不会，因为 Qt 有"父子对象树"机制。** 创建控件时通常会指定一个「父对象（parent）」：

```cpp
auto *label = new QLabel("菜名", card);   // 第二个参数 card 就是父对象
```

规则是：**父对象被销毁时，会自动把它的所有子对象一起销毁。** 只要顶层窗口关掉，整棵树上的控件都会被自动 `delete`，不用我们手动管。

有两种方式让控件挂到父对象上：

1. 构造时直接传 parent：`new QLabel("x", card)`。
2. **把控件 `addWidget` 进某个布局**——它会自动认这个布局所属的控件当父亲（这叫 reparent，自动改认父对象）。所以本项目里很多 `new DishCard(...)` 没传 parent，但 `addWidget` 之后就被托管了。

还有一个特殊函数 `deleteLater()`：在「清空重建列表」时用（见 2.10 后面的套路），它的意思是「**等会儿安全的时候再删**」，避免在正在处理事件时把自己删掉导致崩溃。

## 2.6 样式表 QSS（让界面好看，类似网页的 CSS）

Qt 控件的外观用 **`setStyleSheet("...")`** 设置，语法几乎和网页的 CSS 一样。本项目的「美团风格」全靠它。例子：

```cpp
loginBtn->setStyleSheet(R"(
    QPushButton {
        background: #0085FF; color: #FFFFFF; border: none;
        border-radius: 6px; font-size: 16px; font-weight: bold;
    }
    QPushButton:hover  { background: #0073E6; }
    QPushButton:pressed { background: #0060BF; }
)");
```

- `R"( ... )"` 是 C++ 的**原始字符串**，里面可以随便换行、写引号不用转义，特别适合写这种多行样式。
- `background`、`color`、`border-radius`（圆角）、`font-size` 这些和 CSS 一样。
- **`:hover`**（鼠标悬停）、**`:pressed`**（按下）是「伪状态」，能做出交互效果。

几种选择器在本项目出现：

| 选择器 | 含义 | 例子 |
|---|---|---|
| `QPushButton { }` | 所有按钮 | 通用样式 |
| `#dishCard { }` | objectName 叫 dishCard 的控件 | `setObjectName("dishCard")` 配合使用 |
| `QListWidget::item { }` | 列表里的每一项 | 分类列表的每行 |
| `QPushButton:hover { }` | 悬停状态 | 鼠标移上去变色 |
| `QPushButton[active="true"]` | 自定义属性 active 为 true 时 | 导航栏高亮当前页 |

最后这种「自定义属性」有个坑：用代码改了属性后，样式**不会自动刷新**，必须手动「重新抛光」：

```cpp
btn->setProperty("active", true);       // 改属性
btn->style()->unpolish(btn);            // 卸掉旧样式
btn->style()->polish(btn);              // 重新上样式 → 这才生效
```

导航栏高亮、菜单推荐按钮的选中态，都是用这招实现的。

## 2.7 QString 和 std::string 的区别

C++ 原生的字符串是 `std::string`；Qt 有自己的字符串 `QString`，**Qt 控件只认 `QString`**。两者要互相转换：

```cpp
QString qs = QString::fromStdString(stdStr);  // std::string → QString
std::string s = qs.toStdString();             // QString → std::string
```

`QString` 还有一个超好用的**格式化**方法 `.arg()`，相当于「填空」：

```cpp
QString("月售 %1").arg(dish.sales);                 // %1 被 sales 替换
QString("实付 ¥%1").arg(total, 0, 'f', 2);          // 保留 2 位小数
QString("¥%1").arg(price, 0, 'f', 0);               // 不要小数
```

`%1`、`%2`…是占位符，`.arg(...)` 按顺序填进去。

## 2.8 `Q_OBJECT` 宏 与 moc

凡是要用「信号与槽」的类，**类定义里必须写一行 `Q_OBJECT`**：

```cpp
class LoginPage : public QWidget
{
    Q_OBJECT          // ← 固定写法，必须有
public:
    ...
signals:
    void loginClicked(...);
};
```

它是一个"魔法宏"。Qt 有个叫 **moc（元对象编译器）** 的工具，会在编译前扫描带 `Q_OBJECT` 的头文件，**自动生成信号与槽所需的额外代码**。我们在 CMake 里开了 `set(CMAKE_AUTOMOC ON)`，这个过程就全自动了，我们不用管。**如果忘了写 `Q_OBJECT`，信号槽会连接失败。**

## 2.9 `.ui` 文件 与 AUTOUIC

`ui/src/order_system.ui` 是一个 XML 文件，用 Qt Designer（可视化拖拽工具）生成，描述了主窗口的基本骨架（一个中央区 `centralwidget` + 状态栏 `statusbar`）。

CMake 里开了 `set(CMAKE_AUTOUIC ON)`，编译时会**自动把 `.ui` 翻译成一个 C++ 头文件 `ui_order_system.h`**，里面有个 `Ui_order_system` 类。主窗口构造时调用：

```cpp
ui->setupUi(this);   // 把 .ui 里描述的控件创建出来
```

之后就能用 `ui->centralwidget`、`ui->statusbar` 访问它们。本项目大部分界面是**纯代码手写**的，只有主窗口外壳用了 `.ui`。

## 2.10 事件与 eventFilter

除了信号槽，Qt 还有更底层的「事件」。本项目在导航栏用到一次：让「用户头像区」被点击时弹出菜单。做法是给它装一个**事件过滤器**：

```cpp
m_userPanel->installEventFilter(this);   // 监视 m_userPanel 的所有事件

bool NavBar::eventFilter(QObject *obj, QEvent *event) {
    if (obj == m_userPanel && event->type() == QEvent::MouseButtonPress) {
        // 是用户面板 + 鼠标按下 → 弹出"退出登录"菜单
        ...
        return true;   // 事件已处理
    }
    return QFrame::eventFilter(obj, event);  // 其它情况交还默认处理
}
```

可以理解成「监听这个控件身上发生的一切，挑出我关心的（鼠标按下）来处理」。

## 2.11 定时器 QTimer 与随机数 QRandomGenerator

排队的「自动叫号」用到定时器。`QTimer::singleShot` 的意思是「**等 N 毫秒后，执行一次某个函数**」：

```cpp
QTimer::singleShot(delayMsec, this, &order_system::onAutoAdvance);
```

配合随机数生成 5～20 秒的随机延迟：

```cpp
int delaySec = QRandomGenerator::global()->bounded(5, 21);  // 返回 [5,21) 的随机整数
```

这样排队队列就会每隔几秒自动往前走一位，模拟「前台在叫号」。

## 2.12 网络请求 与 JSON（"智能推荐"会用到）

项目里的「智能推荐」要联网调用 AI 大模型，用到两组以前没讲的东西。

**① 网络请求 `QNetworkAccessManager`（属于 Qt6::Network 模块）**

它负责「替我们去服务器要数据」。关键点是**异步**：发出请求后程序不会卡住等结果，而是**先继续跑**，等结果回来时触发一个信号再处理。

```cpp
QNetworkAccessManager *mgr = new QNetworkAccessManager(this);
QNetworkRequest request{QUrl("https://api.deepseek.com/...")};   // 目标地址
request.setHeader(QNetworkRequest::ContentTypeHeader, "application/json");
QNetworkReply *reply = mgr->post(request, 正文字节);   // 发出 POST 请求，立刻返回一个"回执"
connect(reply, &QNetworkReply::finished, this, [this, reply]() {
    // 这段代码不会马上执行，而是等服务器回复到了才执行
    QByteArray data = reply->readAll();   // 读出服务器返回的内容
    reply->deleteLater();                 // 用完安全删除
});
```

这正是 2.4 讲的「信号与槽」的又一次应用：「网络回复到达」是信号，处理回复的 lambda 是槽。**异步**的好处是请求期间界面不卡死，还能显示「正在请求…」。

**② JSON（`QJsonObject` / `QJsonDocument`）**

JSON 是网络上传数据的通用文本格式，长得像「键值对」。和服务器对话要把数据打包成 JSON 发出去、再把回来的 JSON 拆开：

```cpp
QJsonObject body;                 // 像一个字典
body["model"] = "deepseek-v4-flash";
body["stream"] = false;
QByteArray bytes = QJsonDocument(body).toJson();   // 对象 → JSON 字节串（发出去）

QJsonDocument doc = QJsonDocument::fromJson(data);          // JSON 字节串 → 对象（收回来）
QString text = doc.object()["choices"].toArray()[0]
                  .toObject()["message"].toObject()["content"].toString();  // 层层取值
```

可以把 `QJsonObject` 理解成一个能嵌套的「字典/表格」，`["键"]` 取值，`.toArray()/.toObject()/.toString()` 按类型取出来。

## 3.3 构建脚本 `ui/CMakeLists.txt` 逐行讲

CMake 是「描述怎么编译这个项目」的工具。逐段看：

```cmake
project(order_system LANGUAGES CXX)        # 项目名叫 order_system，语言是 C++
set(CMAKE_PREFIX_PATH "D:/Qt/6.9.3/mingw_64") # ← Qt 安装路径（每台电脑要改成自己的）
set(CMAKE_AUTOUIC ON)                      # 自动处理 .ui 文件（见 2.9）
set(CMAKE_AUTOMOC ON)                      # 自动处理 Q_OBJECT / 信号槽（见 2.8）
set(CMAKE_AUTORCC ON)                      # 自动处理资源文件
set(CMAKE_CXX_STANDARD 17)                 # 用 C++17 标准
find_package(Qt6 COMPONENTS Widgets Network REQUIRED)  # Widgets=界面，Network=智能推荐联网

include_directories(${CMAKE_CURRENT_SOURCE_DIR}/../include)  # 让编译器能找到 include/ 里的头文件

# 自动收集 ui/src 下所有 .cpp
file(GLOB srcs CONFIGURE_DEPENDS ${CMAKE_CURRENT_SOURCE_DIR}/src/*.cpp)

# 不在 ui/src 里的后端实现，要手动一个个加进来
list(APPEND srcs .../storage/FileManager.cpp)        # 数据层
list(APPEND srcs .../order/src/Order_mgr.cpp)        # 购物车/结算
list(APPEND srcs .../queue/src/queue_service.cpp)    # 排队
list(APPEND srcs .../comment/src/comment_service.cpp)# 评论排序

add_executable(${PROJECT_NAME} WIN32 ${srcs})        # 把所有源文件编成一个 exe
target_link_libraries(${PROJECT_NAME} PRIVATE Qt6::Widgets Qt6::Network)  # 链接界面库 + 网络库
```

> 📌 因为加了「智能推荐」要联网，`find_package` 和 `target_link_libraries` 都多了 **Network** 模块，否则用到 `QNetworkAccessManager` 会链接报错。

两个重点：

1. **`file(GLOB ... CONFIGURE_DEPENDS)`**：自动收集 `ui/src/` 下所有 `.cpp`。加了 `CONFIGURE_DEPENDS`，以后**新增或删除 UI 文件，重新编译时会自动重新扫描**，不会出现「新加的文件没被编译、链接报 undefined reference」的坑。
2. **后端 `.cpp` 要手动 `list(APPEND ...)`**：因为它们不在 `ui/src/` 里，`GLOB` 扫不到。**这是答辩可能被问的点**：「新写了一个后端服务怎么让它生效？」答案是「在 CMakeLists 里手动加一行 `list(APPEND srcs ...)`，否则它不会被编译进程序」。

编译运行命令：

```bash
mkdir build && cd build
cmake ../ui -G "MinGW Makefiles"
cmake --build .
./order_system.exe
```

## 3.4 程序的一生（从打开到关闭）

```
main()                                   ← 程序入口（main.cpp）
 └ QApplication a(argc, argv);           ← 创建 Qt 应用（管理整个程序）
 └ order_system w;                       ← 创建主窗口对象
     └ 构造函数:
         ├ ui->setupUi(this);           ← 把 .ui 外壳搭起来
         ├ loadData();                  ← 从 txt 读入菜单/用户/评论，算好推荐榜
         └ setupUI();                   ← 创建所有页面、连好所有信号槽
 └ w.show();                            ← 显示窗口
 └ return a.exec();                     ← 进入"事件循环"：程序停在这里转圈，
                                          不停接收鼠标点击/键盘输入等事件，
                                          直到窗口关闭才退出
```

**「事件循环 `a.exec()`」是 GUI 程序的核心**：程序不是从头跑到尾就结束，而是停在这个循环里**等用户操作**。每次点击、输入都会变成一个「事件」，触发对应的信号槽。这也是为什么我们的逻辑都写在「响应信号」的回调里，而不是写成一条直线流程。

## 4.2 双结构体设计（`Dish` 和 `Dish_qt`）

项目里几乎每种数据有两个版本，定义在 `include/FileManager.h`：

```cpp
struct Dish {            // C++ 版：文字用 std::string，给文件读写/逻辑计算
    int id;  std::string name, description, category;
    double price, rating;  int sales, comment_count;
};
struct Dish_qt {         // Qt 版：文字用 QString，给界面显示
    int id;  QString name, description, category;
    double price, rating;  int sales, comment_count;
};
```

同理还有 `User`/`User_qt`、`CommentMsg`/`CommentMsg_qt` 等。**为什么搞两套？** 因为文件读写、后端算钱用 `std::string` 方便，而 Qt 控件只认 `QString`。两套之间的转换函数集中放在 `FileManager` 里（如 `dish_to_qt`、`dish_to_cpp`），调用示例：

```cpp
Dish d = m_fl.dish_to_cpp(item);   // Qt 版 → C++ 版，交给 OrderService
m_orderService->addDish(d);
```

## 4.3 FileManager 关键函数

`FileManager` 是数据层的全部。几个核心函数：

**`LoadMenu()`** —— 读菜单 + 顺便算好三个推荐榜：
```cpp
while (getline(ifs, line)) {                 // 一行一行读
    if (line.empty() || line[0]=='#') continue; // 跳过空行/注释
    std::stringstream ss(line);
    ss >> d.id >> d.name >> d.price >> ...;  // 按空格拆成各字段
    all_dishes_cpp.push_back(d);             // 存进内存
    ...收集分类...
}
// 读完后，分别按 销量/评分/评论数 排序，各取前 5
std::sort(copy.begin(), copy.end(),
          [](const Dish_qt&a, const Dish_qt&b){ return a.sales > b.sales; });
for (int i=0; i<qMin(5,copy.size()); i++) recommend_by_sales.append(copy[i]);
```
`std::stringstream` + `>>` 是 C++ 里**按空格切分一行**的常用手法。

**`AddCommentAndUpdateMenu(comment, CommentService& cs)`** —— 提交评论时调用。注意它现在多接收一个 `CommentService&` 参数，把「算平均分」这件事**交给 CommentService**，自己只管落盘：
1. 把评论追加到 `comment.txt`；
2. 调用 `cs.AddComment(comment)` —— CommentService 内部更新该菜的平均分和评论数（公式还是「(旧平均分 × 旧评论数 + 新评分) ÷ 新评论数」这个增量平均，只是搬到了 CommentService 里）；
3. 用 `cs.getDishAverRate(菜id)` 把**最新平均分**取回来，同步进内存的 `all_dishes_cpp / all_dishes_qt`；
4. 把更新后的菜单整体写回 `menu.txt`。

```cpp
cs.AddComment(comment);                          // ① 评分计算在 CommentService 里
double newRating = cs.getDishAverRate(dishIdStr);// ② 取回最新平均分
dish.rating = newRating; dish.comment_count++;   // ③ 同步内存 → ④ 写回 menu.txt
```
**好处**：评分计算只有一处（CommentService），评论排序和评分更新用的是同一套数据，不会两边算得不一致。

**序列化 `to_String` / `from_String`** —— 定义在 `QueueMsg`、`CommentMsg` 里，负责「结构体 ↔ 一行文本」互转。比如 `QueueMsg::to_String()` 把三个字段拼成 `"1001|101|1779926400"`，`from_String` 再用 `find('|')` 把它拆回来。

## 4.4 一个细节：static 成员

`FileManager` 里的数据（`all_dishes_cpp` 等）都是 **`static`（静态）成员**，意味着**整个程序共享同一份**，不管创建多少个 `FileManager` 对象，菜单数据只有一份。这就是为什么评论弹窗里临时 `new` 一个 `FileManager` 也能拿到同样的数据。

# 第 5 部分　UI 逐组件精讲（重点）

这一部分把每个界面组件拆开讲。每个组件都遵循同一个模式，先记住这个模式：

> **头文件(.h)** 声明：这个类有哪些**信号**（对外喊什么）、哪些**公开方法**（外面能调什么）、哪些**成员控件**（内部有哪些控件）。

## 5.1 `main.cpp` —— 程序入口

```cpp
int main(int argc, char *argv[]) {
    QApplication a(argc, argv);   // 每个 Qt GUI 程序必须有一个 QApplication
    order_system w;               // 创建主窗口
    w.show();                     // 显示
```

## 5.2 `order_system` 主窗口 —— 总指挥（最重要）

主窗口是整个程序的「大脑」，它**自己几乎不画具体界面**，而是：① 加载数据；② 创建所有页面；③ 把页面用信号连起来、统一处理业务。

**页面用 `QStackedWidget` 管理。** 它像一摞 PPT，5 个页面叠在一起，同时只显示一页：

```cpp
m_stackedWidget->addWidget(m_loginPage);    // index 0 登录
m_stackedWidget->addWidget(m_registerPage); // index 1 注册
m_stackedWidget->addWidget(m_menuPage);     // index 2 菜单
m_stackedWidget->addWidget(m_cartPage);     // index 3 购物车
m_stackedWidget->addWidget(m_queuePage);    // index 4 排队
```

「翻页」统一由 `switchPage(index)` 完成：

```cpp
void order_system::switchPage(int index) {
    if (index == 3 && m_orderService) {        // 进购物车前，先把最新订单刷上去
        m_cartPage->setCart(buildQtOrder(), m_orderService->getDiscountRate(m_currentUser));
    }
    if (index == 4) refreshQueuePage();         // 进排队页前，先刷新排队进度
    m_stackedWidget->setCurrentIndex(index);    // 真正翻页
    m_navBar->setActiveNav(index);              // 导航栏高亮对应按钮
}
```

**所有页面的信号都在 `setupUI()` 里 `connect` 到主窗口**。例如登录：

```cpp
connect(m_loginPage, &LoginPage::loginClicked, this,
        [this](const QString &name, const QString &password) {
    if (checkUser(name, password)) {            // 验证账号密码
        m_orderService = new OrderService(m_currentUser);  // 为该用户建购物车服务
        m_navBar->setUser(m_currentUser);       // 导航栏显示用户信息
        m_navBar->setVisible(true);             // 登录后才显示导航栏
        m_menuPage->setData(...);               // 把菜单数据交给菜单页
        m_menuPage->setDiscountRate(...);       // 把折扣率交给菜单页（影响会员价显示）
        switchPage(2);                          // 跳到菜单页
    } else {
        QMessageBox::warning(this, "登录失败", "用户名或密码不正确！");
    }
});
```

`QMessageBox` 是 Qt 内置的弹窗提示框，一行就能弹出「警告/信息」对话框。

**`buildQtOrder()`** 是一个常用小工具：购物车后端存的是 C++ 版 `Dish`，要显示得转成 `Dish_qt`：

```cpp
QList<Dish_qt> order_system::buildQtOrder() {
    QList<Dish_qt> result;
    for (const auto &d : m_orderService->getOrder())   // 遍历后端订单
        result.append(m_fl.dish_to_qt(d));             // 逐个转 Qt 版
    return result;
}
```

主窗口持有的关键成员（在 `order_system.h`）：5 个页面指针、当前用户 `m_currentUser`、购物车服务 `m_orderService`、排队服务 `m_queueService`、评论服务 `m_commentService`、文件管理器 `m_fl`、取餐号 `m_myQueueId` 等。

## 5.4 `AuthPage`（登录页 + 注册页）

`auth_page.h` 里**一个文件放了两个类**：`LoginPage` 和 `RegisterPage`（之前是分开两个文件，后来合并以减少文件数）。两者长得几乎一样，所以把共用的两个「小工厂函数」抽出来复用——`makeCard` 造白卡片、`makeInput` 造输入框：

```cpp
QLineEdit *makeInput(QWidget *parent, const QString &placeholder, bool password) {

}
```

这种「小工厂函数」避免了重复代码——登录页和注册页都用它造输入框。

**登录页**只发 `loginClicked(name, password)`，验证交给主窗口。**注册页**多一个「确认密码」框，并且在本组件内部做两道校验：**① 用户名是否已存在 ② 两次密码是否一致**，都通过了才发 `registerClicked`：

```cpp
connect(regBtn, &QPushButton::clicked, this, [this]() {
    // ① 读 users.txt，看用户名是否已被占用
    FileManager fl; fl.LoadUsers();
    for (const auto &u : fl.getUsers_cpp())
        if (m_nameEdit->text() == QString::fromStdString(u.name)) {
            QMessageBox::warning(this, "注册失败", "该用户已存在！");
            return;
        }
    // ② 两次密码是否一致
    if (m_pwdEdit->text() != m_pwdConfirmEdit->text())
        QMessageBox::warning(this, "注册失败", "两次输入的密码不相同！");
    else
        emit registerClicked(m_nameEdit->text(), m_pwdEdit->text());
});
```

这是「**表单自己做格式/重复校验、真正的建号交给上层**」的典型分工。

## 5.5 `MenuPage` 菜单页

左右结构：左边 `QListWidget` 分类列表（全部 / 推荐 / 各菜系），右边是推荐切换栏 + 菜品卡片滚动区。

**核心方法 `refreshDishList(category)`** —— 用 2.5 提到的「清空重建」套路刷新卡片：

```cpp
void MenuPage::refreshDishList(const QString &category) {
    // 1) 删掉所有旧卡片
    QLayoutItem *child;
    while ((child = m_dishListLayout->takeAt(0)) != nullptr) {
        if (child->widget()) child->widget()->deleteLater();
        delete child;
    }
    // 2) 按当前分类/推荐方式生成新卡片
    auto makeCard = [&](const Dish_qt &dish) {
        // 查这道菜在销量榜/好评榜的名次（不在榜就是 0）
        int salesRank = 0, ratingRank = 0;
        for (int i=0;i<m_bySales.size();i++)  if (m_bySales[i].id==dish.id){ salesRank=i+1; break; }
        for (int i=0;i<m_byRating.size();i++) if (m_byRating[i].id==dish.id){ ratingRank=i+1; break; }
        auto *card = new DishCard(dish, m_discountRate, salesRank, ratingRank, m_dishContainer);
        connect(card, &DishCard::addClicked,     this, [this](int id){ emit addDishClicked(id); });
        connect(card, &DishCard::commentClicked, this, [this](int dishId){ /* 打开评论弹窗 */ });
        m_dishListLayout->addWidget(card);
    };
    if (category == "推荐") { /* 用销量/评分/评论榜之一 */ }
    else { /* 遍历全部菜，类别匹配的才造卡片 */ }
}
```

**打开评论弹窗**就在这里：点卡片的「查看评论」→ 卡片发 `commentClicked(dishId)` → 菜单页直接弹出 `CommentDialog`：
```cpp
CommentDialog dlg(d, this);
dlg.exec();     // 模态弹窗：弹出后会阻塞，直到用户关掉它
```
`exec()` 是对话框的「**模态**」打开方式——弹窗期间主界面点不了，必须先处理完弹窗。

**推荐栏**：选「推荐」分类时顶部出现**四个**按钮「销量最高 / 评分最高 / 最多评价 / 智能推荐」，点击切换，靠 `switchRecommendMethod` + 自定义属性高亮实现。前三个切换显示对应的榜。

**智能推荐（AI · DeepSeek）** 是这里最特别的功能：选它时会**隐藏菜品卡片**，转而联网请大模型给出推荐文字。核心在 `onSmartRecommend()`（用到 2.12 讲的网络请求 + JSON）：

```cpp
void MenuPage::onSmartRecommend() {
    m_aiLoadingLabel->setVisible(true);          // 先显示"正在请求…"
    QNetworkRequest request{QUrl(DEEPSEEK_API_URL)};
    request.setRawHeader("Authorization", ("Bearer " + DEEPSEEK_API_KEY).toUtf8());
    QJsonObject body;                            // 拼请求 JSON
    body["model"] = "deepseek-v4-flash";
    body["messages"] = /* 系统提示 + buildPrompt() 拼的用户提示 */;
    QNetworkReply *reply = m_networkManager->post(request, QJsonDocument(body).toJson());
    connect(reply, &QNetworkReply::finished, this, [this, reply]() {  // 异步等回复
        reply->deleteLater();
        if (reply->error() != QNetworkReply::NoError) { showAIResponse("网络请求失败…"); return; }
        QJsonDocument doc = QJsonDocument::fromJson(reply->readAll());
        QString content = doc.object()["choices"].toArray()[0]
                             .toObject()["message"].toObject()["content"].toString();
        showAIResponse(content);                 // 把 AI 的话显示到只读文本框
    });
}
```

`buildPrompt()` 负责把「地点（长沙）+ 当前时间星期 + 整张菜单」拼成一段提示词，让 AI 结合时间季节天气推荐 3~5 道菜。

> ⚠️ `menu_page.cpp` 顶部硬编码了 DeepSeek 的 API Key，运行时要联网才能用——答辩演示前确认网络正常；公开代码时这个 Key 应该挪到配置/环境变量（见第 7 部分）。

对外接口：`setData(...)`（喂数据）、`setDiscountRate(rate)`（改折扣并刷新）。对外信号：`addDishClicked(dishId)`、`dishCountChanged(count)`。

## 5.6 `DishCard` 菜品卡片

一道菜一张卡片，是「组件复用」的好例子——菜单上几十张卡片都是同一个类的不同实例。

构造参数很说明问题：
```cpp
DishCard(const Dish_qt &dish, double discountRate, int salesRank, int ratingRank, QWidget *parent);
```
传入这道菜的数据、当前用户折扣率、两个榜单名次。

**会员双价**：折扣率 < 1（是会员）时，显示「灰色删除线原价 + 金橙色会员价」；否则只显示红色原价：
```cpp
if (m_discountRate < 1.0) {
    m_priceLabel = new QLabel(QString("¥%1").arg(dish.price,0,'f',0), this);
    m_priceLabel->setStyleSheet("...text-decoration: line-through...");   // 删除线
    double memberPrice = dish.price * m_discountRate;
    m_memberPriceLabel = new QLabel(QString("会员价：¥%1").arg(memberPrice,0,'f',0), this);
    m_memberPriceLabel->setStyleSheet("...color:#E8960A...");             // 金橙色
}
```

**榜单徽章**：名次 > 0 才显示小标签，销量榜橙色、好评榜绿色：
```cpp
if (m_salesRank > 0)  makeBadge(QString("本店销量第%1").arg(m_salesRank), "#FF6200");
if (m_ratingRank > 0) makeBadge(QString("好评榜第%1").arg(m_ratingRank),  "#52C41A");
```

**图片占位**：没有真实图片，就用一个分类对应颜色的圆角色块 + 菜名首字当占位图（`categoryColor()` 按类别返回颜色）。

对外信号：`addClicked(dishId)`（点了加购）、`commentClicked(dishId)`（点了查看评论）。

## 5.7 `CartPage` 购物车页

**两套界面切换**：购物车空时显示「🛒 购物车是空的 + 去挑选菜品 + 历史订单点菜」；非空时显示「左边菜品列表 + 右边金额汇总」。用 `setVisible(true/false)` 在两个 `QWidget` 之间切换。

**数量统计用 `QMap`**：后端订单是「一份菜一条记录」（点两份就两条），购物车要合并成「× N」显示，所以用 `QMap<int,int>`（菜品 id → 数量）统计：
```cpp
void CartPage::setCart(const QList<Dish_qt> &items, double discountRate) {
    m_counts.clear(); m_dishes.clear();
    for (const auto &it : items) {
        if (!m_counts.contains(it.id)) m_dishes.append(it);  // 第一次见，记下来（保持顺序）
        m_counts[it.id]++;                                   // 计数 +1
    }
    refreshDisplay();
}
```

**右边金额**实时算：商品合计、会员折扣减免、实付款：
```cpp
double discount = subtotal * (1.0 - m_discountRate);   // 省了多少
double total    = subtotal * m_discountRate;           // 实付
```
折扣率 = 1（普通用户）时，折扣行自动隐藏。

对外信号：`increaseRequested`/`decreaseRequested`（加减一份）、`clearCartRequested`（清空）、`checkoutRequested`（结算）、`backToMenuRequested`（继续点餐）、`historyOrderRequested`（历史订单点菜）。每个都对应主窗口里一段处理逻辑。

## 5.8 `CheckoutDialog` 取餐 + 评价弹窗

> ⚠️ 注意：这个弹窗**不是结算时弹的**。现在的流程是「结算 → 拿取餐号去排队 → 在排队页点『取餐』按钮 → 才弹出本窗口评价」，更贴近真实点餐体验（详见 5.10 和功能六）。窗口标题也相应叫「取餐成功」。

上方显示「实付金额 + 取餐号」，中间是「本次菜品标签 + 五星评分 + 文字评价框」，底部「跳过评价 / 提交评价」。

**星级交互**：5 个 `QPushButton` 当星星，点第 n 颗就把前 n 颗点亮；再点同一颗可取消：
```cpp
void CheckoutDialog::onStarClicked(int star) {
    m_rating = (m_rating == star) ? 0 : star;   // 点同一颗→清零，否则设为该值
    updateStars();                              // 重画 5 颗星的颜色
    m_submitBtn->setEnabled(m_rating > 0);      // 没评分时"提交"按钮置灰
}
```
`setEnabled(false)` 把按钮**置灰不可点**，引导用户先打分。

**两个出口**：
- 「跳过评价」→ `reject()`（对话框返回「拒绝」）。
- 「提交评价」→ `accept()`（返回「接受」）。

主窗口在「取餐」时（`onPickup`）据此决定要不要保存评论：
```cpp
// 点了「取餐」→ 取出这一单暂存的信息 → 弹评价窗
PendingReview pr = m_pendingReviews.value(queueId);
CheckoutDialog dlg(pr.dishNames, pr.dishIds, pr.total, queueId, pr.userId, this);
dlg.exec();
CommentMsg cm = dlg.getComment();      // 把用户填的内容组装成 CommentMsg
if (cm.rate > 0) {                     // 有评分才保存
    // 一次搞定：写 comment.txt + 让 CommentService 算新平均分 + 同步内存 + 写回 menu.txt
    m_fl.AddCommentAndUpdateMenu(cm, m_commentService);   // 注意第二个参数（见 4.3）
    m_allItems = m_fl.getMenu_qt(); /* ...刷新榜单... */   // 取回更新后的数据给菜单页
}
```

`getComment()` 负责把界面输入打包成后端的 `CommentMsg`（用户 id、菜品 id 列表、文字、星级、当前时间戳）。

## 5.9 `CommentDialog` 查看评论弹窗

点菜品卡片「查看评论」弹出。结构：标题栏（含**排序下拉框**）→ 评分概览（大号平均分 + 五星 + 星级分布条）→ 评论列表。

**筛选**：构造时 `new` 一个 `FileManager` 加载所有评论，挑出 `dish_ids` 含当前菜 id 的：
```cpp
const std::string idStr = std::to_string(dish.id);
for (const auto &c : allComments)
    for (const auto &did : c.dish_ids)
        if (did == idStr) { m_comments.append(c); break; }
```

**星级分布条**用 `QProgressBar`（进度条）：统计每个星级各有多少条，做成「5★ ▓▓▓ 3」这样的可视化：
```cpp
int dist[6] = {0};
for (const auto &c : comments) if (c.rate>=1 && c.rate<=5) dist[c.rate]++;
for (int star=5; star>=1; star--) makeRatingBar(star, dist[star], total, ...);
```

**排序（功能三的「按评价高低/留言先后」）**：下拉框有**三个选项**——「按时间排序 / 好评优先 / 差评优先」。排序逻辑**不在弹窗里写**，而是统一交给后端 `CommentService::getComment(模式)` 实现（单一来源，避免重复代码）。弹窗只负责选模式、取结果、重建列表：
```cpp
// CommentDialog 构造时：把本菜评论喂进 CommentService
for (const auto &c : 该菜的评论) m_commentService.AddComment(c);

// 切换下拉框时：把界面索引翻译成模式字符串，调 getComment 取排好序的结果
std::string rankType = "time";                     // 0 = 按时间
if (m_sortMode == 1)      rankType = "rate_high";  // 1 = 好评优先
else if (m_sortMode == 2) rankType = "rate_low";   // 2 = 差评优先
m_comments.clear();
for (const auto &c : m_commentService.getComment(rankType)) m_comments.append(c);
refreshCommentList();   // 清空并按新顺序重建卡片
```
而 `CommentService::getComment` 内部就是三段 `std::sort`：
```cpp
if (rank_type == "time")       sort(... a.in_time > b.in_time ...);  // 时间大=更新，在前
else if (rank_type=="rate_high")sort(... a.rate > b.rate ...);        // 高分在前
else if (rank_type=="rate_low") sort(... a.rate < b.rate ...);        // 低分在前
```
这样三种排序方式集中在一个函数里，弹窗代码更干净。

## 5.10 `QueuePage` 排队进度页

结构：顶部「当前叫号」大号横幅 + 我的取餐号状态 → 中间左右两列「预约排队中 / 待取餐」→ 底部只有「返回菜单」（进度会自动刷新，不需要手动刷新按钮）。

**它是纯展示页**：自己不操作排队队列，只通过 `setQueueData(...)` 接收快照来显示，操作逻辑全在主窗口（这又是「页面只显示、主窗口做事」的体现）。

```cpp
void QueuePage::setQueueData(int currentCall,
                             const std::vector<QueueMsg> &waiting,
                             const std::vector<QueueMsg> &taking,
                             int myQueueId) {
    m_currentCall = currentCall; m_waiting = waiting;
    m_taking = taking; m_myQueueId = myQueueId;
    refreshDisplay();   // 清空重建两列
}
```

**我的号高亮**：遍历两列时判断 `queue_id == myQueueId`，是自己的就用橙色边框+橙色号码牌，并在顶部提示「前面还有 N 桌 / 已叫号请取餐 / 已取餐」。

**「取餐」按钮**：右侧「待取餐」列里，每张票不再是「请取餐」文字标签，而是一个橙色的「**取餐**」按钮。点击后发出 `pickupRequested(取餐号)` 信号，由主窗口弹出评价窗、并把这个号移出取餐队列（见下方 `onPickup`）：
```cpp
auto *takeBtn = new QPushButton("取餐", row);
const int qid = msg.queue_id;
connect(takeBtn, &QPushButton::clicked, this, [this, qid]() {
    emit pickupRequested(qid);   // 把"哪个号取餐了"喊给主窗口
});
```

**等待时长**：用 `(当前时间 - 入队时间) / 60` 算分钟，显示「刚刚下单 / 已等待 N 分钟」。

对外信号：`backToMenuRequested`（返回菜单）、`pickupRequested(int queueId)`（点某张票的「取餐」）。进度刷新由主窗口在「切到本页 / 自动叫号 / 取餐」时主动调 `setQueueData` 完成，所以不需要「刷新」信号。

**自动叫号在主窗口**，用定时器（见 2.11）：结算后 `scheduleAutoAdvance()` 启动，每 5～20 秒随机触发一次 `onAutoAdvance`，把预约队列第一位移到取餐队列：
```cpp
void order_system::onAutoAdvance() {
    m_autoAdvancePending = false;
    if (!m_queueService.is_Empty()) {
        m_queueService.advance_queue();  // 队首：预约 → 取餐，当前叫号 +1
        refreshQueuePage();              // 刷新界面
    }
    scheduleAutoAdvance();               // 安排下一次
}
```
`m_autoAdvancePending` 是个开关，防止重复启动多个定时器。

**取餐处理也在主窗口**（`onPickup`）：用户点某张待取餐票的「取餐」→ 主窗口先弹评价窗（这一单还没评价过的话），保存评论，再调用 `m_queueService.take_meal(取餐号)` 把它移出取餐队列、刷新界面：
```cpp
void order_system::onPickup(int queueId) {
    if (m_pendingReviews.contains(queueId)) {   // 这单还存着待评价信息
        // ...弹 CheckoutDialog、保存评论（见 5.8）...
        m_pendingReviews.remove(queueId);       // 评价过/跳过都不再重复弹
    }
    m_queueService.take_meal(queueId);          // 取餐：移出取餐队列
    refreshQueuePage();
}
```
取餐后这张票从「待取餐」消失，顶部状态自动变成「已取餐，感谢惠顾」。

# 第 6 部分　七大功能端到端串讲

前面是「按组件」拆，这一部分是「按功能」把 UI → 服务 → 数据**串成一条线**，方便你给老师演示时讲清「点了这一下，背后发生了什么」。

## 功能一：菜单分类

`FileManager::LoadMenu()` 读 `menu.txt` 时顺手收集所有「类型」字段 → 主窗口把分类列表交给 `MenuPage::setData()` → 左侧 `QListWidget` 显示分类 → 点某分类触发 `onCategoryChanged` → `refreshDishList(分类)` 只为该类菜生成 `DishCard`。

## 功能二：折扣计算 + 会员升级（结算全链路）

这是最长的一条链，建议重点演示：

```
菜单点"+"  → DishCard 发 addClicked → 主窗口 m_orderService->addDish(d)
                                       （后端 order_ 里 push 一条）
进购物车   → switchPage(3) → setCart(...) → CartPage 用 QMap 合并数量、算实付
点"去结算" → CartPage 发 checkoutRequested → 主窗口:
   ① total = m_orderService->checkout()
        checkout 内部 = calcCart() × getDiscountRate(等级)
        然后 updateUser(): 写 history_order.txt、menu.txt 销量+1、
                           users.txt 总消费累加并重判等级
   ② 重新 LoadUsers，把最新等级同步回 m_currentUser 和 OrderService
   ③ m_queueService.in_queue() 入队拿取餐号；启动自动叫号
   ④ 把本单菜品信息暂存进 m_pendingReviews（等取餐时再评价）
   ⑤ 弹「结算成功」提示（金额+取餐号），跳到排队页 switchPage(4)
   —— 评价不在这里弹了，挪到了"取餐"时（见功能三/功能六）
```

**会员等级规则**（在 `OrderService` 里）：

| 等级 | 升级门槛(总消费) | 折扣 |
|---|---|---|
| REGULAR | <1000 | 1.00 |
| SILVER | ≥1000 | 0.95 |
| GOLD | ≥5000 | 0.85 |
| PLATINUM | ≥10000 | 0.75 |

折扣由 `getDiscountRate()` 按等级返回；升级判定在 `updateUser()` 里：消费累加后用门槛重新判等级写回文件。

## 功能三：点餐评价（写 + 看 + 排序）

- **写**：在排队页点「取餐」时弹 `CheckoutDialog` 打分写评价 → `AddCommentAndUpdateMenu(评论, m_commentService)` 把评论存进 `comment.txt`，由 `CommentService` 算出该菜新平均分，再同步内存并写回 `menu.txt` → 主窗口取回更新后的数据刷新菜单页，新评分立刻在菜单和评论里生效。
- **看**：菜品卡片「查看评论」→ `CommentDialog` 筛出该菜评论、算平均分、画星级分布条。
- **排序**：评论弹窗右上角下拉框「按时间 / 好评优先 / 差评优先」三选一 → 调 `CommentService::getComment(模式)` 取排好序的结果 → 重建列表（三种排序都在 getComment 里实现）。

## 功能四：推荐点餐（Top5 + 徽章）

`FileManager::LoadMenu()` 读完菜单立即按销量、评分、评论数各排序取前 5，存成三个推荐榜 → 菜单页选「推荐」分类时，顶部四个按钮切换显示哪个榜 → 同时 `DishCard` 在生成时查出自己在销量/好评榜的名次，**在卡片上挂「本店销量第 N / 好评榜第 N」徽章**。这样「最好的 5 个菜的排序属性额外显示在菜单中」就实现了。

**智能推荐（AI）扩展**：第四个按钮「智能推荐」走的是另一条路——`MenuPage::onSmartRecommend()` 用 `QNetworkAccessManager` 异步 POST 到 DeepSeek 大模型，请求体里塞进「菜单 + 当前时间 + 长沙地点」的提示词，AI 回一段「结合时间季节天气」的推荐文字，显示在只读文本框里（见 2.12 和 5.5）。这是在「需求四」基础上的加分项。

## 功能五：点餐记忆（复制历史订单）

购物车空状态点「历史订单点菜」→ `CartPage` 发 `historyOrderRequested` → 主窗口 `OrderService::loadUserHistoryOrders()` 按**用户名**从 `history_order.txt` 读出该用户所有历史单 → 弹对话框列出（菜名用「、」连接）→ 用户选一条「加入购物车」→ 把该单中**当前菜单里仍存在**的菜逐个 `addDish` 回购物车（已下架的菜会提示「X 道菜当前菜单中不存在」）→ 跳到购物车页。

## 功能六：排队显示

```
结算成功 → m_queueService.in_queue(订单号) 入"预约队列"，返回取餐号(从1001起)
         → 暂存本单信息到 m_pendingReviews → scheduleAutoAdvance() 启动定时器 → 跳排队页
每5~20秒 → onAutoAdvance() → advance_queue()：预约队列队首 → 取餐队列，当前叫号+1
进排队页 → switchPage(4) → refreshQueuePage() → QueuePage::setQueueData() 显示快照
（进度自动刷新：切到本页、每次自动叫号、取餐后，主窗口都会再 refreshQueuePage() 一次）
点某票"取餐" → 发 pickupRequested(号) → onPickup()：弹评价窗 + take_meal(移出取餐队列) + 刷新
```

「预约排队进度」= 预约队列里你前面还有几桌；「取餐排队进度」= 你的号是否已进入取餐队列。两者都在 `QueuePage` 实时显示，自己的号高亮。点「取餐」后该票出队、状态变「已取餐」，并触发评价窗——这样「评价」就和「取餐」这个真实动作绑在了一起。

## 功能七：系统界面

整套界面统一「美团 / 饿了么」视觉：灰底（`#F5F5F5`）、白色圆角卡片、蓝色主操作（`#0085FF`）、橙红价格（`#FF4D2E`）、金色会员价（`#E8960A`）。全部用 QSS 样式表实现（见 2.6）。

# 第 8 部分　验收答辩问答预演

下面把老师**最可能问的问题**按类别整理，并给出参考回答。建议每位组员**至少熟悉自己负责模块**对应的问题。回答时要点：先一句话结论，再点出对应的类/函数，能说出文件名最好。

## A 类　Qt 基础（最常被问，因为这是课程重点）

**Q1：信号与槽是什么？你们项目怎么用的？**
> 信号与槽是 Qt 的事件通信机制：某个控件发生事情（信号），自动触发我们绑定的处理代码（槽），用 `connect` 连接。我们项目用它把**界面和业务逻辑解耦**：每个页面只在用户操作时 `emit` 一个自定义信号（比如登录页发 `loginClicked`），主窗口 `connect` 接住后才去执行验证、跳转等逻辑。好处是改界面不动逻辑、多人协作不冲突。

**Q2：代码里全是 `new`，为什么几乎没有 `delete`？会内存泄漏吗？**
> 不会。Qt 有「父子对象树」：创建控件时指定父对象，或把它 `addWidget` 进布局后会自动认父对象。父对象销毁时会自动销毁所有子对象。我们的控件最终都挂在主窗口这棵树上，主窗口关闭时整棵树自动析构。需要中途删除时（比如刷新列表）我们用 `deleteLater()` 安全删除。

**Q3：界面布局是怎么排的？用了绝对坐标吗？**
> 没有用绝对坐标，用的是 Qt 的布局管理器 `QVBoxLayout`（竖排）和 `QHBoxLayout`（横排），它们自动计算控件位置、自动随窗口缩放。复杂界面靠布局**嵌套**实现，比如菜品卡片就是「水平布局里套竖直布局」。`addStretch()` 加弹簧把控件推向一边。

**Q4：界面这么好看，样式怎么做的？**
> 用 Qt 样式表 QSS，语法类似网页 CSS，通过 `setStyleSheet()` 设置颜色、圆角、字号、悬停效果等。比如按钮的 `:hover`、`:pressed` 伪状态做交互反馈。导航栏高亮用自定义属性 `[active="true"]` + `unpolish/polish` 刷新实现。

**Q5：`Q_OBJECT` 是什么？为什么每个类都写它？**
> 它是启用信号槽机制的宏。Qt 的 moc（元对象编译器）会扫描带 `Q_OBJECT` 的类，自动生成信号槽需要的底层代码。我们在 CMake 里开了 `AUTOMOC` 自动完成。不写它信号槽就用不了。

**Q6：`QString` 和 `std::string` 有什么区别？为什么项目里两个都有？**
> `std::string` 是 C++ 标准字符串，`QString` 是 Qt 的字符串，Qt 控件只接受 `QString`。我们后端逻辑、文件读写用 `std::string` 方便，界面显示用 `QString`，所以每种数据设计了两个版本（如 `Dish` / `Dish_qt`），转换函数集中在 `FileManager`。

**Q7：模态对话框 `exec()` 和普通窗口 `show()` 有什么区别？**
> `show()` 显示后程序继续往下跑；`exec()` 是模态打开，弹窗期间会阻塞、主界面点不了，直到用户关掉弹窗，并返回「接受/拒绝」结果。我们的结算弹窗、评论弹窗都用 `exec()`，因为需要等用户处理完再继续。

## B 类　架构与设计（体现思考深度）

**Q8：项目整体架构是怎样的？为什么这么分？**
> 三层：界面层（各页面 + 主窗口）、服务层（OrderService/QueueService/CommentService 管业务规则）、数据层（FileManager 管文件读写）。每层只跟相邻层打交道。好处是职责清晰、便于分工、易于替换——比如以后想把 txt 换成数据库，只改数据层。

**Q9：主窗口 `order_system` 起什么作用？**
> 它是「总指挥」，自己不画具体界面，负责三件事：启动时 `loadData()` 读数据、`setupUI()` 创建所有页面并用 `connect` 把信号接起来、统一处理所有业务逻辑（登录验证、结算、入队等）。页面之间不直接通信，都经过主窗口中转。

**Q10：5 个页面怎么切换的？**
> 用 `QStackedWidget`，它像一摞 PPT，5 页叠在一起同时只显示一页。`switchPage(index)` 负责翻页并高亮导航栏对应按钮。索引 0 登录、1 注册、2 菜单、3 购物车、4 排队。

**Q11：为什么每种数据要做 C++ 版和 Qt 版两个结构体？不嫌麻烦吗？**
> 是有点重复，但换来界面和后端的彻底分离：后端完全不依赖 Qt（纯 C++，可单独测试），界面拿到的都是能直接显示的 `QString`。转换逻辑只集中在 `FileManager` 一处，可控。

## C 类　数据存储

**Q12：数据为什么不用数据库，用 txt？**
> 这是教学项目，重点练 C++ 和 Qt，不是数据库。txt 方案简单、人能直接打开看、方便演示和检查。代价是没有并发和事务，但单机小项目够用。架构上数据层是隔离的，以后换数据库不影响上层。

**Q13：一行文本怎么解析成结构体？**
> 空格分隔的（menu/users）用 `std::stringstream` + `>>` 逐字段读取；竖线分隔的（comment/queue）用 `string::find('|')` 找分隔符再 `substr` 切片。每个消息结构体还提供 `to_String`/`from_String` 做「结构体 ↔ 文本行」的序列化。

**Q14：评论提交后，菜品的平均分是怎么更新的？**
> 在 `AddCommentAndUpdateMenu` 里用增量公式：`新平均分 = (旧平均分 × 旧评论数 + 本次评分) / (旧评论数 + 1)`，然后把这道菜在 `menu.txt` 里的整行重写回去，评论数也 +1。提交后我们重新 `LoadMenu`，新分数立刻在界面生效。

**Q15：菜名里如果有空格会怎样？**
> 会解析错位，因为 menu.txt 用空格分字段。我们的约定是菜名和描述不含空格。这是已知限制，改进方法是换成别的分隔符（如竖线）。

## D 类　各功能实现细节

**Q16：会员折扣怎么算？等级怎么升的？**
> 折扣率由 `OrderService::getDiscountRate()` 按等级返回（普通1.0 / 白银0.95 / 黄金0.85 / 白金0.75）。结算时 `checkout` 用「合计 × 折扣率」算实付，再在 `updateUser` 里把实付累加到总消费，按门槛（1000/5000/10000）重新判定等级写回 `users.txt`。升级后跳回菜单会用新折扣率刷新会员价。

**Q17：推荐的「最好的 5 个菜」怎么选出来的？**
> 在 `LoadMenu` 读完菜单后，用 `std::sort` 分别按销量、评分、评论数排序，各取前 5 名存成三个榜。菜单页「推荐」分类可切换看哪个榜；同时每张卡片会查自己在榜里的名次，挂上「本店销量第 N / 好评榜第 N」徽章。

**Q17b：「智能推荐」是怎么实现的？真的接了 AI 吗？（重点加分项）**
> 是真的接了大模型。点「智能推荐」后，`MenuPage::onSmartRecommend()` 用 Qt 的网络模块 `QNetworkAccessManager` 向 DeepSeek 的接口发一个 HTTP POST 请求，请求体是用 `QJsonObject` 拼的 JSON，里面包含模型名和一段提示词——提示词由 `buildPrompt()` 把「整张菜单 + 当前时间星期 + 长沙地点」拼起来，让 AI 结合时间、季节、天气推荐 3~5 道菜。请求是**异步**的：发出后界面先显示「正在请求…」，等服务器回复触发 `QNetworkReply::finished` 信号，再用 `QJsonDocument` 把返回的 JSON 解析出来、取出 `choices[0].message.content` 显示。需要联网、`CMakeLists` 里链接了 `Qt6::Network`。可以补一句：API Key 目前硬编码在源码里，正式发布应挪到配置文件。

**Q18：排队的号是怎么自动往前走的？**
> 用 `QTimer::singleShot` 定时器。结算入队后启动，每隔 5～20 秒随机触发一次 `onAutoAdvance`，调用 `advance_queue()` 把预约队列队首移到取餐队列、当前叫号 +1，然后刷新界面，并安排下一次。模拟前台叫号。`m_autoAdvancePending` 开关防止重复启动定时器。

**Q19：历史订单复制是怎么做的？为什么有的菜显示"不存在"？**
> `loadUserHistoryOrders` 按用户名从 `history_order.txt` 读出历史单，弹窗让用户选一条，把其中**当前菜单里仍存在**的菜（按菜名匹配）加回购物车。如果某道菜已从菜单下架，就匹配不到，提示「X 道菜当前菜单中不存在」，只加能加的。

**Q20：评论能按什么排序？怎么实现的？**
> 评论弹窗右上角下拉框有三种：「按时间」（最新在前）、「好评优先」（评分高→低）、「差评优先」（评分低→高）。三种排序**统一在 `CommentService::getComment(模式)` 里用 `std::sort` 实现**，弹窗只负责把下拉框索引翻译成模式字符串、调用它取结果、清空重建列表。这样排序逻辑只有一处，便于维护。

## E 类　C++ / 数据结构

**Q21：项目里用到哪些容器和算法？**
> `std::vector`（菜单、用户、订单、队列）、`std::map`（CommentService 的菜品→评论映射）、Qt 的 `QList`（界面用的列表）、`QMap`（购物车数量统计）。算法主要是 `std::sort` 排序（推荐榜、评论排序）和线性查找（按 id/名字找菜）。

**Q22：lambda 表达式 `[this](){...}` 是什么？**
> 是 C++11 的匿名函数（就地定义的小函数）。`[this]` 是「捕获列表」，表示函数体里能访问当前对象的成员。我们大量用它作为信号的处理代码，写在 `connect` 里，不用单独定义一堆槽函数。

**Q23：`static` 成员变量为什么用在 FileManager？**
> `FileManager` 的数据成员是 static，全程序共享一份。这样不管在哪里临时 `new` 一个 FileManager，拿到的都是同一份菜单/用户数据，不会重复加载。

## F 类　项目管理与拔高

**Q24：你负责哪一块？遇到的最大难点是什么？**
> （按实际分工答）举例：「我负责排队模块。难点是排队要『实时更新』，但桌面程序没有真实的后厨在叫号，我用 `QTimer` 定时器模拟自动叫号，并把界面做成纯展示、逻辑放主窗口，保证刷新时数据一致。」

**Q25：这个项目你觉得做得好的地方？**
> ① 清晰的三层架构和信号槽解耦，界面与逻辑分离；② 完整复刻了美团风格 UI；③ 七个需求功能全部跑通，还做了会员价、榜单徽章、星级分布、取餐评价等细节；④ **额外接入了 AI 智能推荐**（联网调用 DeepSeek 大模型，按时间天气推荐菜品），是个亮眼的加分项；⑤ 构建脚本用 `GLOB CONFIGURE_DEPENDS` 避免了加文件忘编译的坑。

**Q26：如果再给一周，你会改进什么？**
> 把排队持久化到文件、让好评榜推荐也走 `CommentService::getBest5Dishs`（评论排序已经统一走它了）、修掉 `addUser` 的写入格式问题、给密码加哈希、把按空格分隔的文件格式改得更健壮。

**Q27（现场改需求）：老师让你现在加一个新菜系分类，怎么做？**
> 直接在 `menu.txt` 里加几行该类别的菜（类型字段写新分类名），不用改代码——`LoadMenu` 会自动收集新分类、菜单左侧会自动出现。这正体现了数据驱动的好处。

**Q28（现场改需求）：把白金折扣从 7.5 折改成 7 折，改哪里？**
> 改一处：`order/src/Order_mgr.cpp` 里 `OrderService::getDiscountRate()` 中 PLATINUM 返回的 `0.75` 改成 `0.70`。重新编译即可，界面会自动按新折扣显示会员价。

**Q29（现场改需求）：想让排队支持手动点一下"叫号"，怎么加？**
> 在 `QueuePage` 加一个按钮和一个 `advanceRequested` 信号，在主窗口 `connect` 到 `m_queueService.advance_queue()` 再 `refreshQueuePage()`。其实之前就是手动按钮，后来改成了定时器自动叫号。注意「叫号」（`advance_queue`：预约→取餐）和「取餐」（`take_meal`：移出取餐队列，由待取餐票上的『取餐』按钮触发）是两个不同动作。

**Q30：演示时如果时间不够，重点演示哪条线？**
> 演示「登录 → 点菜 → 购物车看会员折扣 → 结算拿取餐号 → 跳到排队页看号自动前进 → 进入待取餐后点『取餐』→ 弹窗打星评价 → 回菜单看评分已更新」这一条主线，能一次串起功能一二三四六七，最有说服力。

**Q31：为什么把评价放到"取餐"时，而不是结算后立刻弹？**
> 更贴近真实点餐：现实中是「下单付款 → 排队等叫号 → 取到餐 → 才评价」。所以我们结算时只把订单信息暂存进 `m_pendingReviews`（按取餐号作 key），等用户在排队页点了对应票的「取餐」按钮，`onPickup` 再取出信息弹评价窗，评价后把该号 `take_meal` 出队。每单只弹一次。
